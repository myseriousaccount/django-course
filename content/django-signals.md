# Сигнали (signals): реакція на події

Сигнали — це спосіб сказати Django: «коли станеться певна подія, виконай ось цей код», не зшиваючи ці два шматки логіки напряму. Цей урок пояснює, що таке сигнал, які події Django надсилає, як на них підписатися і — головне — коли сигнали доречні, а коли краще без них. Приклади з **різних доменів** (акаунти, магазин, розсилка), щоб механізм не здавався прив'язаним до одного випадку.

## Що таке сигнал

> **Сигнал** — це повідомлення «щось сталося», яке один шматок коду **надсилає**, а інший — **отримує** й реагує. Модель об'єкта, що надіслав подію, нічого не знає про тих, хто на неї підпишеться: зв'язок непрямий.

Ідея в одному рядку: **подія → обробник**. Django сам розсилає низку вбудованих сигналів на типові події життєвого циклу об'єктів — збереження, видалення, зміна зв'язків. Ти лише пишеш функцію-обробник і кажеш, на який сигнал вона реагує.

```
об'єкт зберігається → Django надсилає сигнал post_save → спрацьовує твій обробник
```

Важлива суть: обробник викликається **сам**, ти його ніде явно не кличеш. Це і сила (код роз'єднаний), і небезпека (важче простежити, «звідки воно взялося») — до цього ще повернемось.

> <i class="bi bi-lightbulb"></i> Аналогія: сигнал — як сповіщення на телефоні. Застосунок, що надіслав пуш, не знає, що ти зробиш у відповідь — може, відкриєш, може, проігноруєш. Джерело події й реакція на неї — незалежні.

## Головні вбудовані сигнали

Django надсилає багато сигналів, але в реальній роботі найчастіше зустрінеш ці п'ять — усі про життєвий цикл моделі:

| Сигнал | Коли надсилається |
|---|---|
| `pre_save` | **перед** збереженням об'єкта в БД |
| `post_save` | **після** збереження (найуживаніший) |
| `pre_delete` | **перед** видаленням об'єкта |
| `post_delete` | **після** видалення |
| `m2m_changed` | коли змінюється зв'язок «багато-до-багатьох» (додали/прибрали теги, ролі…) |

Різниця `pre_` / `post_` проста: `pre_` — коли ще можна **втрутитися** в об'єкт до запису (наприклад, дописати поле), `post_` — коли об'єкт **уже** в базі й можна робити побічні дії (створити пов'язаний запис, надіслати лист).

## Аргументи обробника

Обробник — звичайна функція. Django передає їй іменовані аргументи, тому в кінці **завжди** ставлять `**kwargs` — щоб не зламатись, якщо Django додасть ще щось:

```python
def my_handler(sender, instance, created, **kwargs):
    ...
```

| Аргумент | Що це |
|---|---|
| `sender` | клас моделі, що надіслав сигнал (`User`, `Order`…) |
| `instance` | сам об'єкт, який зберегли/видалили |
| `created` | тільки в `post_save`: `True`, якщо об'єкт **щойно створено**, `False` — якщо оновлено |
| `**kwargs` | решта (обов'язково лови, навіть якщо не використовуєш) |

> <i class="bi bi-pin-angle"></i> `created` — ключова деталь `post_save`. Без перевірки `if created:` твій код спрацює і при **кожному** оновленні об'єкта, а не лише при першому створенні.

## Як підписатися: `@receiver` + `apps.py`

Класична схема з трьох кроків. Обробники живуть в окремому файлі `app/signals.py`, а «вмикають» їх у конфізі застосунку.

**Крок 1.** Файл `accounts/signals.py` — сам обробник:

```python
from django.db.models.signals import post_save
from django.dispatch import receiver
from django.contrib.auth.models import User
from .models import Profile

@receiver(post_save, sender=User)          # реагуй на post_save моделі User
def create_profile(sender, instance, created, **kwargs):
    if created:                            # лише коли User щойно створено
        Profile.objects.create(user=instance)
```

**Крок 2.** `accounts/apps.py` — метод `ready()` імпортує файл із сигналами, щоб Django його «побачив»:

```python
from django.apps import AppConfig

class AccountsConfig(AppConfig):
    default_auto_field = 'django.db.models.BigAutoField'
    name = 'accounts'

    def ready(self):
        import accounts.signals          # реєструє всі @receiver у файлі
```

**Крок 3.** Переконайся, що app підключений у `INSTALLED_APPS` саме через свій конфіг (`'accounts'` цього достатньо — Django знайде `AccountsConfig` сам).

> <i class="bi bi-exclamation-triangle"></i> Найпоширеніша причина «сигнал не спрацьовує» — забутий імпорт у `ready()`. Декоратор `@receiver` нічого не реєструє, доки Python **не виконає** файл `signals.py`. Саме `import accounts.signals` у `ready()` це й робить. Не клади імпорт на початок `apps.py` (буде помилка «apps aren't loaded yet») — тільки всередині `ready()`.

## Приклад 1 — Profile після реєстрації User (класика)

Найтиповіший випадок узагалі. Вбудована модель `User` — **чужа**, ти не можеш дописати в її метод `save()` свою логіку. Сигнал — природний вихід: щойно створюється користувач, автоматично з'являється його профіль.

```python
@receiver(post_save, sender=User)
def create_profile(sender, instance, created, **kwargs):
    if created:
        Profile.objects.create(user=instance)
```

## Приклад 2 — списати склад після замовлення

Магазин: коли створюється `OrderItem`, зменшуємо залишок товару.

```python
from django.db.models.signals import post_save
from django.dispatch import receiver
from .models import OrderItem

@receiver(post_save, sender=OrderItem)
def reduce_stock(sender, instance, created, **kwargs):
    if created:
        product = instance.product
        product.stock -= instance.quantity
        product.save()
```

## Приклад 3 — вітальний лист новому користувачу

Розсилка: реагуємо на ту саму подію (`post_save` для `User`), але робимо зовсім інше. Це й ілюструє силу сигналів — **кілька незалежних реакцій** на одну подію, кожна у своєму app.

```python
from django.core.mail import send_mail

@receiver(post_save, sender=User)
def send_welcome_email(sender, instance, created, **kwargs):
    if created:
        send_mail(
            'Вітаємо!',
            'Дякуємо за реєстрацію.',
            'noreply@example.com',
            [instance.email],
        )
```

## Коли сигнали НЕ потрібні

Сигнали спокушають, але це **«магія на відстані»**: логіка спрацьовує «сама», і новачок у коді може довго шукати, звідки взявся профіль чи лист. Тому правило таке:

- <i class="bi bi-exclamation-triangle"></i> **Якщо подія й реакція — у твоєму власному коді і поруч**, часто зрозуміліше зробити дію **явно**: у методі моделі (`save()`), у формі або в сервісній функції. Читач бачить усе в одному місці, без «прихованих» обробників.
- <i class="bi bi-info-circle"></i> **Сигнали доречні, коли:**
  - джерело події **чуже** — вбудований `User`, стороння модель, куди не можна дописати `save()`;
  - реакцій **багато** й вони **незалежні** (створити профіль + надіслати лист + записати лог) — розкидати їх окремими обробниками чистіше, ніж роздути один `save()`;
  - реакція справді **побічна** й не критична для основної операції.

> <i class="bi bi-lightbulb"></i> Просте мірило: якщо дію легко й природно вписати прямо в `save()` своєї моделі — впиши її туди. Сигнал бери, коли дописати нікуди (чужа модель) або коли реакцій стало забагато для одного місця.

## Типові помилки / Нюанси

> <i class="bi bi-exclamation-triangle"></i> **Забутий імпорт у `ready()`** — обробник просто мовчки не спрацьовує. Перевір `import <app>.signals` всередині `ready()`.

> <i class="bi bi-exclamation-triangle"></i> **Забутий `if created:`** — код на `post_save` виконається і при кожному оновленні об'єкта. Для «зробити раз при створенні» перевірка `created` обов'язкова.

> <i class="bi bi-exclamation-triangle"></i> **Подвійне спрацювання** — якщо обробник `post_save` сам викликає `instance.save()`, він знову надішле `post_save` → ризик нескінченного циклу. Або уникай `save()` всередині, або оновлюй через `Model.objects.filter(pk=...).update(...)`, який сигналів **не** надсилає.

> <i class="bi bi-info-circle"></i> `.update()` на QuerySet і **bulk-операції** (`bulk_create`, `bulk_update`) сигналів `pre_save`/`post_save` **не** викликають. Якщо логіка мусить спрацювати — оновлюй об'єкти по одному або винеси її явно.

> <i class="bi bi-pin-angle"></i> Обробник виконується **синхронно**, у тому ж запиті. Довгу дію (надсилання листа, звернення до API) краще винести в фонову задачу, щоб не гальмувати відповідь користувачу.

## Підсумок

- **Сигнал** — непряма реакція «подія → обробник»; джерело події не знає про обробників.
- Головні: `pre_save`, `post_save`, `pre_delete`, `post_delete`, `m2m_changed`. `pre_` — втрутитися до запису, `post_` — побічні дії після.
- Обробник: `@receiver(post_save, sender=Model)`, аргументи `sender, instance, created, **kwargs`; `created` розрізняє створення й оновлення.
- Підключення: обробники в `app/signals.py`, а `import app.signals` — усередині `AppConfig.ready()`.
- **Не** зловживай: якщо дію можна явно вписати у свою модель — впиши. Сигнали — для чужих моделей (`User`) і для багатьох незалежних реакцій.
- Стережись подвійного спрацювання (`save()` в обробнику) і пам'ятай, що `.update()`/bulk сигналів не шлють.
