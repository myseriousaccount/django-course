# Філософія Django: суть і принципи

Перш ніж писати код, варто зрозуміти, **на яких принципах побудований Django**. Це не теорія заради теорії: коли ти бачиш логіку за рішеннями фреймворку, його структура перестає здаватися набором довільних правил і стає передбачуваною. Цей урок дає загальну карту — далі ми розберемо кожну її частину окремо.

## Що таке Django (коротко)

> **Django** — це full-stack вебфреймворк мовою Python, побудований за принципом «batteries included» («батарейки в комплекті»): типові потреби вебдодатку — робота з базою даних, маршрутизація, шаблони, автентифікація, адмін-панель — уже вбудовані.

Простими словами: Django дає не цеглу, а вже зведений каркас будинку. Твоє завдання — облаштувати його під конкретний проєкт, а не закладати фундамент щоразу з нуля.

## Девіз, що пояснює характер

Неофіційний девіз Django — **«The web framework for perfectionists with deadlines»** («вебфреймворк для перфекціоністів із дедлайнами»).

У ньому закладено компроміс, який визначає весь дизайн фреймворку:

- **perfectionists** — рішення мають бути чистими й правильними;
- **deadlines** — і водночас швидкими у втіленні.

Django не змушує обирати між «якісно» і «швидко»: він пропонує готові, перевірені рішення, тож ти будуєш надійно, але не витрачаєш час на велосипеди.

## Чотири принципи, на яких стоїть Django

### 1. DRY — Don't Repeat Yourself («не повторюйся»)

**Визначення.** Одну одиницю знання описують **в одному місці**, а все похідне фреймворк виводить із неї автоматично.

**Як це працює.** Канонічний приклад — модель як єдине джерело правди. Ти описуєш сутність один раз:

```python
class Product(models.Model):
    name = models.CharField(max_length=200)
    price = models.DecimalField(max_digits=10, decimal_places=2)
```

…і з цього **одного** опису Django виводить структуру таблиці в БД (через міграції), форму для введення даних, інтерфейс в адмін-панелі та правила валідації. Ти не дублюєш опис товару в чотирьох місцях — він існує в одному.

**Навіщо.** Дублювання — головне джерело помилок: зміниш в одному місці, забудеш у другому. DRY усуває цей клас проблем у корені.

### 2. Конвенція над конфігурацією (convention over configuration)

**Визначення.** Фреймворк діє за **усталеними стандартами за замовчуванням**, тому налаштовувати доводиться лише те, що відхиляється від стандарту.

**Як це працює.** Структуру папок, розташування шаблонів, іменування файлів Django вже визначив за тебе. Команди `startproject` і `startapp` створюють «правильний» каркас автоматично.

> <i class="bi bi-pin-angle"></i> «Над», а не «замість». Конфігурація нікуди не зникає — налаштувати все одно можна. Просто конвенція працює як розумний дефолт, тож конфігуруєш лише виняткові випадки.

**Зворотний бік.** Конвенції треба **знати**: якщо назвати файл не за стандартом або покласти шаблон не в ту папку, автоматизм не спрацює. Саме тому структуру ми вивчаємо окремою темою.

### 3. Явне краще за неявне (explicit is better than implicit)

**Визначення.** Залежності й дані передаються відкрито, а не «з'являються» з прихованого глобального контексту. Це принцип із «Дзену Python», якого Django послідовно дотримується.

**Як це працює.** Приклад, який ти вже бачила, — об'єкт запиту передається у view **явним аргументом**:

```python
def index(request):     # request видно одразу — він приходить параметром
    ...
```

Django міг би зробити `request` глобальним (як це влаштовано у Flask), але обрав явність: дивлячись на сигнатуру функції, ти одразу знаєш, з чим вона працює.

**Навіщо.** Менше прихованого стану — простіше читати, тестувати й налагоджувати код.

### 4. Слабка зв'язність, сильна цілісність (loose coupling)

**Визначення.** Компоненти системи максимально незалежні один від одного, а пов'язана логіка зібрана разом.

**Як це працює.** Django ділить проєкт на **застосунки (apps)** — самодостатні модулі. Шари теж розділені: рівень даних (Model) не знає про HTML (Template), тож зміна шаблону не зачіпає логіку.

**Навіщо.** Незалежні частини легше розвивати, тестувати й переносити. Добре спроєктований app можна підключити до іншого проєкту майже без змін.

## Як це збирається разом: патерн MTV

Усі принципи працюють у межах архітектурного патерну **MTV** (Model–Template–View) — Django-варіанта класичного MVC:

```
Запит → URLconf → View (логіка) → Model (дані) + Template (HTML) → Відповідь
```

| Шар | Відповідає за |
|---|---|
| **Model** | дані та правила їх обробки |
| **Template** | те, як дані показуються користувачу |
| **View** | логіку, що поєднує дані з відображенням |

Детально кожен шар розберемо в наступних темах — поки достатньо бачити загальну схему руху запиту.

## Карта ключових понять

Ці терміни зустрічатимуться далі. Зараз їх не треба заучувати — це легенда до карти, щоб слова не лякали наперед:

| Поняття | Що це | Аналог із Flask |
|---|---|---|
| **Проєкт** | весь сайт цілком (один на застосунок) | весь додаток |
| **App** | модуль під одну функціональну область (їх багато) | blueprint |
| **Model** | опис даних = таблиця БД | модель SQLAlchemy |
| **ORM** | робота з БД через Python-об'єкти | Flask-SQLAlchemy |
| **Migration** | контрольована зміна структури БД | Flask-Migrate |
| **View** | логіка обробки запиту | view-функція |
| **Template** | HTML-шаблон сторінки | шаблон Jinja2 |
| **URLconf** | таблиця маршрутів | `@app.route` |
| **Middleware** | проміжні обробники навколо кожного запиту | before/after_request |
| **Admin** | готова панель керування даними | (реалізовувалась вручну) |
| **Settings** | конфігурація всього проєкту | `app.config` |

## Зміна підходу після Flask

| Аспект | Flask | Django |
|---|---|---|
| Структура | визначаєш сама | задана конвенціями |
| Обсяг із коробки | мінімум, решту збираєш | майже все одразу |
| Підхід | «склади під свої потреби» | «використовуй готове за стандартом» |
| Ціна | багато рішень на тобі | потрібно знати конвенції |

Жоден підхід не «кращий» — вони для різних задач. Але практичний висновок при переході такий: **не борися з конвенціями, а вивчай їх** — саме в них і захована швидкість Django.

## Підсумок

- **Django** — full-stack фреймворк за принципом «batteries included»; девіз «для перфекціоністів із дедлайнами» означає «якісно **і** швидко».
- Чотири принципи: **DRY** (єдине джерело правди), **конвенція над конфігурацією** (розумні дефолти), **явність** (`request` як аргумент), **слабка зв'язність** (незалежні apps і шари).
- Усе працює в межах патерну **MTV**: Model — дані, Template — відображення, View — логіка.
- Головний зсув після Flask: структуру не вигадуєш, а вивчаєш конвенції — і це пришвидшує розробку.
