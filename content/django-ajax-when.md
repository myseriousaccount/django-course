# AJAX: коли і навіщо

Питання «коли саме треба AJAX? для запитів до БД?» виникає майже в усіх — і в основі його лежить один поширений міф. Розвіймо його одразу, а потім розберемо, де AJAX справді доречний, зберемо цілий каталог сценаріїв із **різних доменів**, пройдемо один приклад наскрізь — і з'ясуємо, чим JSON відрізняється від HTML-фрагмента у відповіді.

## Спершу — розвіяти міф

AJAX — це **не «спосіб робити запити до бази даних»**. До бази звертається Django-**view** через ORM — і при звичайному завантаженні сторінки, і при AJAX-запиті. Тобто база тут узагалі ні до чого: різниця не в ній.

<i class="bi bi-lightbulb"></i> Уяви офіціанта. Він однаково ходить на кухню по твоє замовлення — і коли ти прийшов уперше, і коли догукав його вже за столиком. «Кухня» (БД) працює однаково. Різниця лише в тому, **встав ти з-за столика чи ні**. AJAX — це якраз про «не вставати».

## Що таке AJAX насправді

> **AJAX** — це спосіб **звернутися до сервера без перезавантаження сторінки** й оновити лише її частину. Ключове слово тут — «без перезавантаження».

**Як це працює.** Замість того щоб браузер завантажив нову сторінку цілком, JavaScript тихо надсилає запит у фоні, отримує невелику відповідь і підставляє її в потрібне місце — решта сторінки лишається на екрані незмінною.

**Навіщо.** Щоб дія відчувалася миттєвою й не «мигала»: користувач лишається там, де був, а оновлюється тільки той шматочок, який реально змінився.

## Порівняння: звичайний запит vs AJAX

Найкраще видно на схемі. Ланцюг **однаковий** в обох випадках:

```
браузер → Django view → (ORM → БД) → відповідь
```

Різниця тільки у **відповіді** та в тому, що з нею робить браузер:

| | Звичайний запит | AJAX |
|---|---|---|
| Що робить браузер | **повне перезавантаження** всієї сторінки | тихий запит у фоні |
| Що оновлюється | вся сторінка | **лише шматочок** |
| Що повертає view | цілу HTML-сторінку | маленький JSON або HTML-фрагмент |
| Робота з БД | однакова (view + ORM) | однакова (view + ORM) |

Тобто клік по звичайному посиланню → уся сторінка перезавантажилась. Клік з AJAX → нічого не перезавантажилось, змінилась лише кнопка чи один блок.

## Пряма відповідь на «для запитів до БД?»

Ні. БД працює **однаково** в обох випадках — її обслуговує view через ORM незалежно від того, AJAX це чи ні.

AJAX обирають з іншої причини: коли ти **не хочеш перезавантажувати сторінку** заради невеликого оновлення. Це рішення про **інтерфейс**, а не про доступ до даних.

## Каталог сценаріїв: де AJAX справді доречний

Лайк — лише найвідоміший приклад, тому його часто вважають єдиним. Насправді сценаріїв багато, і в кожному сенс той самий: **оновити частину, не перезавантажуючи все**. Ось карта типових ситуацій з **різних доменів** — щоб ти впізнавала їх у власному проєкті.

- **Жива пошук книг (бібліотека).** Користувач друкує «пай…» у полі — під ним одразу з'являються книги, назва яких містить цей текст. Навіщо тут AJAX: результати мають оновлюватися **на кожне натискання клавіші**, а перезавантажувати сторінку на кожну літеру було б безглуздо.
- **«Завантажити ще» / безкінечний скрол (блог).** Показано 12 статей, унизу кнопка «Показати ще». Натиснув — доклеїлись наступні 12, ти лишився там само. Навіщо: дотягнути **наступну порцію** даних, не втрачаючи позицію прокрутки й уже завантажені картки.
- **Залежні списки (жанр → фільм).** Обрав у першому списку «Комедія» — другий список сам наповнився фільмами цього жанру. Навіщо: другий список залежить від першого, і його вміст треба **підтягнути з сервера** після вибору, без перезавантаження форми.
- **Перевірка доступності username під час набору (реєстрація).** Поки користувач вводить логін, поряд з'являється «вільний ✓» або «зайнятий ✗». Навіщо: перевірити унікальність може лише сервер (він знає БД), а показати відповідь треба **до відправлення форми**. Це прямий місток до теми валідації — сервер валідовує, AJAX лише доставляє відповідь миттєво.
- **«Додати в кошик» без перезавантаження (магазин).** Натиснув «У кошик» — лічильник у шапці став `3`, кнопка змінилась на «У кошику». Навіщо: підтвердити дію й оновити **лічильник та кнопку**, не виривати користувача зі сторінки каталогу.
- **Оцінка / рейтинг (кінотека).** Поставив фільму 8 із 10 — середня оцінка одразу перерахувалась. Навіщо: відправити голос і показати **новий результат** на місці, без переходу на іншу сторінку.
- **Інлайн-редагування (блог).** Клікнув по заголовку своєї статті в кабінеті — він перетворився на поле вводу, змінив, натиснув Enter — збереглося. Навіщо: редагувати **один запис прямо в списку**, без окремої сторінки-форми.

Спільне в усіх пунктах: після дії користувач **лишається на місці**, а змінюється лише невеликий фрагмент. Це і є ознака, за якою впізнають AJAX.

## Наскрізний приклад: жива пошук книг

Розберемо один сценарій повністю — від Django-view до JavaScript у шаблоні. Візьмемо живу пошук книг у бібліотеці: користувач друкує в полі, результати з'являються під ним.

**Крок 1. View, що повертає JSON.** Замість HTML-сторінки він віддає список знайдених книг у форматі JSON.

```python
from django.http import JsonResponse
from .models import Book

def search_books(request):
    query = request.GET.get("q", "")          # текст із поля пошуку
    books = Book.objects.filter(title__icontains=query)[:10]
    results = [
        {"id": b.id, "title": b.title, "year": b.published_year}
        for b in books
    ]
    return JsonResponse({"results": results})
```

**Крок 2. Маршрут** у `urls.py`:

```python
path("books/search/", views.search_books, name="search_books"),
```

**Крок 3. JavaScript у шаблоні**, що читає поле й вставляє результати:

```html
<input type="text" id="search" placeholder="Пошук книг...">
<ul id="results"></ul>

<script>
  const input = document.getElementById("search");
  const list = document.getElementById("results");

  input.addEventListener("input", () => {
    fetch(`/books/search/?q=${encodeURIComponent(input.value)}`)
      .then((response) => response.json())
      .then((data) => {
        list.innerHTML = "";                        // очищаємо попередні
        data.results.forEach((book) => {
          const li = document.createElement("li");
          li.textContent = `${book.title} (${book.year})`;
          list.appendChild(li);
        });
      });
  });
</script>
```

Що тут відбувається: на кожне натискання клавіші `fetch` тихо питає в'ю, в'ю через ORM (`Book.objects.filter`) шукає в БД і повертає JSON, а JavaScript будує з нього `<li>` і вставляє в список. Сторінка не перезавантажується жодного разу.

> <i class="bi bi-info-circle"></i> **Чому тут немає CSRF-токена?** Це **GET**-запит — він лише читає дані й нічого не змінює. CSRF-захист потрібен тільки для запитів, які **щось змінюють** на сервері (POST, PUT, DELETE). Пошук нічого не змінює, тож токен не потрібен.

Порівняй із дією, що **змінює** дані, — «поставити лайк статті» в блозі. Це вже POST, і тут токен обов'язковий:

```javascript
fetch(`/posts/42/like/`, {
    method: "POST",
    headers: { "X-CSRFToken": csrftoken },
})
    .then((response) => response.json())
    .then((data) => {
        button.textContent = data.liked ? "♥ Подобається" : "♡ Подобається";
        counter.textContent = data.likes_count;
    });
```

```python
def toggle_like(request, post_id):
    post = Post.objects.get(pk=post_id)
    # ... логіка додавання/зняття лайка через ORM ...
    return JsonResponse({"liked": True, "likes_count": post.likes.count()})
```

> <i class="bi bi-exclamation-triangle"></i> Для **POST**-запитів через AJAX обов'язковий **CSRF-токен** (`X-CSRFToken`). Без нього Django відхилить запит із помилкою 403 — і це правильно, це захист від підробки запитів. Просте правило: **GET читає — токен не треба; POST змінює — токен обов'язковий.**

## AJAX повертає JSON чи HTML?

У наскрізному прикладі view віддавав JSON, а розмітку будував JavaScript. Але це не єдиний шлях. Є два підходи — і вибір між ними важливий.

**Підхід 1: view віддає JSON, JavaScript будує розмітку.**

View повертає «чисті дані» (`JsonResponse`), а зібрати з них HTML (`<li>`, картку, кнопку) — робота JavaScript.

- <i class="bi bi-plus-circle"></i> Гнучко: ті самі дані можна показати як завгодно, зручно для складних інтерфейсів.
- <i class="bi bi-dash-circle"></i> Розмітку доводиться дублювати в JS — вона живе окремо від Django-шаблонів.

**Підхід 2: view віддає готовий HTML-фрагмент.**

View рендерить маленький шаблон і повертає **вже готовий шматок HTML**, а JavaScript просто вставляє його як є:

```python
from django.shortcuts import render

def search_books(request):
    query = request.GET.get("q", "")
    books = Book.objects.filter(title__icontains=query)[:10]
    return render(request, "library/_results.html", {"books": books})
```

```javascript
fetch(`/books/search/?q=${encodeURIComponent(input.value)}`)
    .then((response) => response.text())          // .text(), не .json()!
    .then((html) => {
        list.innerHTML = html;                     // вставляємо готовий HTML
    });
```

- <i class="bi bi-plus-circle"></i> Простіше: розмітку пишеш у звичному Django-шаблоні (`_results.html`), а не дублюєш у JS.
- <i class="bi bi-dash-circle"></i> Менш гнучко, якщо тим самим даним потрібні різні відображення.

> <i class="bi bi-info-circle"></i> Для багатьох задач підхід із HTML-фрагментом простіший, бо не змушує писати розмітку двічі. Саме на ньому побудований інструмент **htmx** — окрема бібліотека, яка дозволяє робити AJAX майже без JavaScript, прямо в атрибутах HTML. Це тема окремого уроку; тут достатньо знати, що такий шлях існує.

## Безпека й права — завжди на сервері

Важливий наголос: AJAX **нічого не змінює** в тому, хто має право на дію. Перевіряти права, автентифікацію й коректність даних мусить **view** — так само, як для звичайного запиту.

```python
from django.contrib.auth.decorators import login_required

@login_required
def toggle_like(request, post_id):
    # сюди не потрапить неавторизований користувач,
    # навіть якщо він надішле AJAX-запит вручну
    ...
```

AJAX-запит — це такий самий HTTP-запит, який будь-хто може підробити з консолі браузера чи скриптом. Тому «сховати» кнопку в інтерфейсі **недостатньо**: якщо view не перевірить права, дію виконає хто завгодно. Правило залізне: **клієнту не довіряй, усі перевірки — у view**. AJAX лише прибирає перезавантаження сторінки, а не рівень безпеки.

## Коли НЕ треба AJAX

**Навіщо.** Щоб не ускладнювати те, що прекрасно працює звичайним способом:

- **звичайна навігація між сторінками** — перехід на сторінку книги чи статті це просте посилання `<a href>`, не AJAX;
- **звичайні форми, де перезавантаження нормальне** — реєстрація, оформлення замовлення: перейти на нову сторінку тут логічно;
- **первинний контент сторінки** — його одразу рендерить Django-шаблон. Не тягни його AJAX-ом «щоб було»: сторінка тільки повільніше покаже порожнечу, а потім доклеїть вміст.

> <i class="bi bi-info-circle"></i> **Правило вибору.** Якщо після дії весь сенс — оновити маленьку частину й **лишитися на місці**, це AJAX. Якщо логічно перейти на іншу сторінку або оновити все — це звичайний запит чи `redirect`.

## Типові помилки / Нюанси

- **Робити AJAX там, де досить звичайного посилання чи форми.** Перехід на іншу сторінку — це `<a href>` або `redirect`, а не fetch. AJAX тут лише додає коду й багів на рівному місці.
- **Думати, що AJAX «швидший доступ до БД».** Ні: база та сама, view та сама, ORM та сама. AJAX не прискорює запит до БД — він лише прибирає перезавантаження сторінки.
- **Забути CSRF-токен у POST.** Найчастіша причина 403 при першому AJAX-запиті. GET токена не потребує, POST — обов'язково.
- **Плутати `.json()` і `.text()`.** Якщо view повертає JSON — читаєш `response.json()`; якщо готовий HTML-фрагмент — `response.text()`. Переплутаєш — отримаєш помилку розбору.
- **Покладатися на «сховану» кнопку замість перевірки у view.** Приховати елемент в інтерфейсі — не захист. Права перевіряє **сервер**, бо AJAX-запит легко підробити вручну.
- **Тягнути AJAX-ом первинний контент.** Те, що видно одразу при відкритті сторінки, має рендерити Django-шаблон. AJAX — для того, що змінюється **після** дії користувача.

## Підсумок

- AJAX — це **не про базу даних**: до БД однаково звертається view через ORM і при звичайному запиті, і при AJAX.
- AJAX — це спосіб **звернутися до сервера без перезавантаження сторінки** й оновити лише її частину.
- Ланцюг `браузер → view → (ORM → БД) → відповідь` однаковий; різниця тільки у відповіді: ціла HTML-сторінка проти маленького JSON чи HTML-фрагмента.
- Сценаріїв багато — жива пошук книг, «завантажити ще» статті, залежні списки жанр→фільм, перевірка username, кошик, рейтинг, інлайн-редагування — і в усіх сенс один: **оновити частину, лишитися на місці**.
- **Правило:** треба лишитися на місці й оновити дрібницю → AJAX; треба перейти / оновити все → звичайний запит або `redirect`.
- Технічно: `fetch` → view повертає `JsonResponse` (дані) **або** HTML-фрагмент через `render` (простіше; згадай htmx). GET — без токена, POST — з **CSRF-токеном**.
- **Безпека — завжди у view:** AJAX не змінює прав; клієнту не довіряй, усі перевірки на сервері.
