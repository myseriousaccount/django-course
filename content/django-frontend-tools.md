# jQuery і популярні фронтенд-інструменти

У попередньому уроці ми розібрали, **навіщо** Django-проєкту JavaScript. Тут — про конкретні **інструменти**, які найчастіше підключають готовими, щоб не писати все з нуля. Історично головний із них — **jQuery**, тож почнемо з нього, а далі — огляд «що зазвичай беруть».

## Що таке jQuery

> **jQuery** — це JavaScript-бібліотека, що спрощує роботу з елементами сторінки (DOM), події та AJAX. Її девіз — *«write less, do more»* («пиши менше, роби більше»). Уся її «магія» ховається за символом **`$`**.

**Як це працює.** Порівняй ту саму дію на чистому JS і на jQuery:

```js
// чистий JavaScript
document.getElementById('menu').classList.toggle('hidden');

// jQuery — коротше
$('#menu').toggleClass('hidden');
```

Тобто jQuery — це не «інша мова», а **зручна обгортка** над звичайним JS.

## Навіщо вона з'явилась і чи потрібна сьогодні

**Навіщо.** Колись браузери поводилися по-різному, а чистий JS був багатослівним і незручним. jQuery згладжувала ці відмінності й давала короткий синтаксис — тому свого часу стояла майже на кожному сайті.

**Сьогодні.** Сучасний JavaScript і однакова поведінка браузерів зробили **більшість** можливостей jQuery зайвими — те саме тепер легко робиться «ванільним» JS. Для **нових** проєктів частіше беруть vanilla JS, **Alpine.js** або **htmx** (з попереднього уроку).

Але jQuery **досі всюди**, і знати її варто, бо ти неминуче з нею зустрінешся:
- у безлічі вже наявних проєктів;
- багато готових **плагінів** (Select2, DataTables та ін.) написані під jQuery;
- старий **Bootstrap 3/4** залежить від неї.

> <i class="bi bi-exclamation-triangle"></i> Не додавай jQuery в новий проєкт «за звичкою». Бери її лише коли: (1) потрібен плагін, що її вимагає, або (2) ти працюєш у наявному коді, де вона вже є. Заради одного `toggleClass` тягнути цілу бібліотеку не варто.

## Підключення

Через CDN — один рядок:

```html
<script src="https://code.jquery.com/jquery-3.7.1.min.js"></script>
```

> <i class="bi bi-exclamation-triangle"></i> Порядок важливий: jQuery має бути підключена **раніше** за будь-який плагін чи твій код, що її використовує. Спершу `<script>` jQuery, потім усе інше.

## Базові патерни (щоб читати чужий код)

- `$(function () { ... })` — виконати код, коли сторінка (DOM) готова.
- Селектори — як у CSS: `$('#id')`, `$('.class')`.

| Дія | jQuery | Чистий JS |
|---|---|---|
| Текст елемента | `$('#x').text('Привіт')` | `el.textContent = 'Привіт'` |
| Значення поля | `$('#x').val()` | `el.value` |
| Клас | `$('#x').addClass('active')` | `el.classList.add('active')` |
| Показати/сховати | `$('#x').hide()` / `.show()` | `el.style.display = 'none'` |
| Клік | `$('#x').on('click', fn)` | `el.addEventListener('click', fn)` |

Видно, що це просто коротший запис звичних дій.

## jQuery + Django: AJAX і CSRF

jQuery має свої скорочення для AJAX — `$.get()`, `$.post()`, `$.ajax()`. Але тут спрацьовує **та сама пастка, що й із `fetch`**: Django відхилить `POST` без CSRF-токена (403).

Класичне рішення для Django — один раз налаштувати токен для всіх запитів:

```js
// прочитати cookie csrftoken
function getCookie(name) {
    const value = `; ${document.cookie}`;
    const parts = value.split(`; ${name}=`);
    if (parts.length === 2) return parts.pop().split(';').shift();
}

// підставляти токен у КОЖЕН ajax-запит
$.ajaxSetup({
    headers: { 'X-CSRFToken': getCookie('csrftoken') }
});

// тепер POST працює — лайк під статтею блогу
$('#like').on('click', function () {
    $.post($(this).data('url'), function (data) {
        $('#like').text(data.liked ? '♥' : '♡');
    });
});
```

А Django-бік — звичайний view, що повертає `JsonResponse` (як в уроці про JS).

## Які «штуки» зазвичай підключають до Django

Ось типовий набір готових інструментів, які беруть замість написання власного коду. Підключають так само — через CDN (`<script>`/`<link>`) або кладуть у `static/`.

| Інструмент | Для чого | Потребує jQuery? |
|---|---|---|
| **Bootstrap** | CSS-фреймворк: сітка, кнопки, форми, модалки | ні (v5), так (v3–4) |
| **jQuery** | DOM, події, AJAX | — |
| **Select2** | «розумні» випадні списки з пошуком | так |
| **DataTables** | таблиці із сортуванням/пошуком/пагінацією | так |
| **Chart.js** | графіки й діаграми | ні |
| **Flatpickr** | зручний вибір дати (календар) | ні |
| **SweetAlert2** | красиві модалки й підтвердження | ні |
| **Toastr** | тост-повідомлення (спливні сповіщення) | так |
| **Font Awesome / Bootstrap Icons** | іконки | ні |
| **Alpine.js / htmx** | легка інтерактивність (сучасна альтернатива) | ні |

> <i class="bi bi-info-circle"></i> Для типового контентного сайту (блог, каталог, фільмотека) достатньо: **Bootstrap** (розмітка й стилі) + набір **іконок**, а за потреби — **Flatpickr** (дати виходу/публікації), **Chart.js** (статистика переглядів), **SweetAlert2** (підтвердження видалення). Це закриває 90% потреб без власного важкого JS.

## Типові помилки / Нюанси

> <i class="bi bi-exclamation-triangle"></i> Плагін не працює — найчастіша причина: його `<script>` стоїть **вище** за jQuery. Порядок завжди: спершу jQuery, потім плагіни, потім твій код.

> <i class="bi bi-exclamation-triangle"></i> Забути CSRF у jQuery-AJAX → 403, як і з `fetch`. Налаштуй `$.ajaxSetup` один раз.

> <i class="bi bi-info-circle"></i> Не змішуй усе підряд. Обирай **або** jQuery-екосистему, **або** сучасну (Alpine/htmx) — двоє відразу роздувають сторінку без потреби.

## Підсумок

- **jQuery** — бібліотека-обгортка над JS (символ `$`), що спрощує DOM/події/AJAX; історично стоїть майже всюди.
- Для **нових** проєктів частіше беруть vanilla JS / **Alpine.js** / **htmx**; jQuery — коли потрібен її плагін або це наявний код.
- **Порядок скриптів:** спершу jQuery, потім плагіни.
- **CSRF** для jQuery-AJAX — через `$.ajaxSetup` із `X-CSRFToken` (та сама пастка, що з `fetch`).
- Типовий набір готових інструментів: **Bootstrap** (CSS) + **іконки** + за потреби Select2/DataTables/Chart.js/Flatpickr/SweetAlert2.

> <i class="bi bi-book"></i> Офіційний сайт — jquery.com; документація Django про CSRF з AJAX — «How to use Django's CSRF protection» (docs.djangoproject.com).
