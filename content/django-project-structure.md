# Структура проєкту: що обов'язкове

На відміну від Flask, де ти сама вигадувала структуру, Django **генерує її за тебе** командою `startproject`. Розберемо, що саме з'являється, навіщо потрібна кожна частина, і що всередині кожного обов'язкового файлу.

## Проєкт ≠ app

**Визначення.** **Проєкт (project)** — це весь сайт цілком; він один. **App (модуль/застосунок)** — окрема функціональна частина проєкту; їх багато.

Це найважливіше розрізнення в Django, і його варто засвоїти одразу:

- **Проєкт** — папка з налаштуваннями (у shop-app це `root/`). Один на сайт.
- **App** — окремий модуль під одну область. Наприклад для блог-платформи це могли б бути `blog`, `users`, `comments`; для кіно-довідника — `movies`, `actors`, `reviews`; для бібліотеки — `catalog`, `members`, `loans`.

> <i class="bi bi-lightbulb"></i> Аналогія: **проєкт — це будинок, apps — це кімнати**. Будинок має спільні комунікації (налаштування, головний роутер), а кожна кімната має своє призначення. У кіно-довіднику одна «кімната» — фільми, інша — актори, третя — рецензії.

## Структура проєкту (на прикладі shop-app)

Ось що реально лежить у shop-app після `startproject root .` і кількох `startapp` — це показує **справжню структуру робочого проєкту**:

```
shop-app/
└── root/                      ← коренева папка проєкту
    ├── manage.py              ← ОБОВ'ЯЗКОВО: пульт керування
    ├── db.sqlite3             ← файл бази (з'являється після migrate)
    │
    ├── root/                  ← ОБОВ'ЯЗКОВО: пакет-проєкт (налаштування)
    │   ├── __init__.py        ← робить папку Python-пакетом
    │   ├── settings.py        ← головні налаштування
    │   ├── urls.py            ← головний роутер
    │   ├── wsgi.py            ← вхідна точка для деплою (синхронна)
    │   └── asgi.py           ← вхідна точка для деплою (асинхронна)
    │
    ├── home/                  ← app (модуль)
    ├── catalog/               ← app
    ├── carts/                 ← app
    ├── accounts/              ← app
    ├── orders/                ← app
    │
    ├── templates/             ← спільні HTML-шаблони
    └── static/                ← CSS / JS / картинки
```

> <i class="bi bi-exclamation-triangle"></i> Зверни увагу на **дві папки `root`**: зовнішня — просто коренева тека проєкту, а внутрішня `root/root/` — це **пакет з налаштуваннями** (бо проєкт назвали `root`). Внутрішню створив `startproject`.

## Що всередині кожного обов'язкового файлу

`startproject` створює п'ять обов'язкових файлів. Розберемо кожен:

**`manage.py`** — скрипт-«пульт» у корені. Через нього виконують усі команди (`runserver`, `migrate`, `startapp`). Знає про твої налаштування, бо містить рядок `os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'root.settings')`.

**`root/__init__.py`** — порожній файл, що робить папку `root/` Python-пакетом (щоб можна було писати `from root import settings`).

**`root/settings.py`** — серце проєкту. Тут: перелік `INSTALLED_APPS`, підключення БД (`DATABASES`), шаблони (`TEMPLATES`), мова й часовий пояс, статика, `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`, посилання на головний роутер (`ROOT_URLCONF`).

**`root/urls.py`** — головний роутер. Сюди приходить кожен запит, звідси його переадресовують у потрібний app через `include()`.

**`root/wsgi.py` / `root/asgi.py`** — точки входу для бойового сервера. WSGI — для синхронних серверів (Gunicorn, uWSGI), ASGI — для асинхронних (Uvicorn, підтримка WebSocket). У розробці не чіпаєш; потрібні при деплої.

## Що всередині app (модуля)

Коли робиш `startapp blog`, Django генерує таку папку:

```
blog/
├── __init__.py
├── admin.py          ← реєстрація моделей в адмінці
├── apps.py           ← конфіг цього app
├── models.py         ← моделі (таблиці БД): напр. Post, Comment
├── views.py          ← view-функції/класи (логіка)
├── tests.py          ← тести
└── migrations/       ← історія змін схеми БД
    └── __init__.py
```

Зверни увагу: `urls.py` тут **немає** — його додають вручну, коли app обслуговує маршрути. Так само за потреби додають `forms.py`, `templates/`, `static/`.

## Що обов'язкове, а що — на твій розсуд

| Файл / папка | Обов'язковість | Навіщо |
|---|---|---|
| `manage.py` | **Обов'язково** | Запуск усіх команд (runserver, migrate…) |
| `<проєкт>/settings.py` | **Обов'язково** | Налаштування всього проєкту |
| `<проєкт>/urls.py` | **Обов'язково** | Головний роутер — без нього немає маршрутів |
| `<проєкт>/wsgi.py` / `asgi.py` | **Обов'язково** | Точка входу для бойового сервера |
| `<проєкт>/__init__.py` | **Обов'язково** | Робить папку Python-пакетом |
| хоча б один app | Практично завжди | Без модулів сайт нічого не робить |
| `templates/` | Якщо рендериш HTML | Місце для шаблонів |
| `static/` | Якщо є CSS/JS | Статичні файли |
| app-івський `urls.py` | Якщо app має маршрути | Локальний роутер модуля |
| `forms.py` | Якщо є форми | Опис і валідація форм |

Перші п'ять файлів Django створює сам — їх **не треба** створювати руками. App-и додаєш командою `startapp` (наступний розділ).

## Звідки Django знає про твою структуру

**Як це працює.** Структура працює не «магічно» — усе зав'язано на `settings.py`. Показано на прикладі уявної блог-платформи з app-ами `blog`, `users`, `comments`:

```python
# settings.py
ROOT_URLCONF = 'root.urls'              # де шукати головний роутер
WSGI_APPLICATION = 'root.wsgi.application'

INSTALLED_APPS = [
    # ... вбудовані django.contrib.* ...
    'blog',                             # ← перелік твоїх apps
    'users',
    'comments',
]

TEMPLATES = [{
    # ...
    'DIRS': [BASE_DIR / 'templates'],   # де лежать спільні шаблони
    'APP_DIRS': True,                   # + шукати шаблони всередині кожного app
}]
```

Тобто папки самі по собі нічого не означають — вони «вмикаються», коли прописані в `settings.py`.

> <i class="bi bi-exclamation-triangle"></i> **Типова помилка.** Створив app командою `startapp`, але не додав його в `INSTALLED_APPS` → Django його не побачить (моделі не мігруються, шаблони не знаходяться, адмінка порожня).

## Папка templates і чому підпапка за модулем

Шаблони кладуть у спільну `templates/` з підпапками за іменем модуля. Приклад для блог-платформи:

```
templates/
├── _layouts/base.html         ← базовий шаблон (від нього наслідуються інші)
├── blog/
│   ├── post_list.html
│   └── post_detail.html
└── users/
    ├── login.html
    └── profile.html
```

**Навіщо підпапка за модулем.** Django шукає шаблон за іменем у **всіх** app-ах підряд. Якщо і `blog`, і `users` матимуть просто `index.html`, Django візьме перший знайдений — конфлікт. Підпапка (`blog/`, `users/`) робить ім'я унікальним. Тому у view пишуть повний шлях: `render(request, 'blog/post_list.html', ...)`.

## Типові помилки / Нюанси

> <i class="bi bi-exclamation-triangle"></i> **App не в `INSTALLED_APPS`** → моделі не мігруються, шаблони «не знаходяться», в адмінці порожньо. Перше, що перевіряти при дивних збоях нового app.

> <i class="bi bi-exclamation-triangle"></i> **Дві однойменні папки `root/`** — це нормально й очікувано, коли проєкт назвали `root`. Зовнішня — коренева тека, внутрішня — пакет-налаштування.

> <i class="bi bi-info-circle"></i> **`urls.py` в app немає за замовчуванням** — його додають вручну. `startapp` генерує лише `models.py`, `views.py`, `admin.py`, `apps.py`, `tests.py`, `migrations/`.

## Підсумок

- **Проєкт** (один, пакет-налаштування) ≠ **app** (багато: blog, users, comments…). Будинок і кімнати.
- `startproject` створює обов'язковий кістяк: `manage.py` + пакет-проєкт із `settings.py`, `urls.py`, `wsgi.py`, `asgi.py`, `__init__.py`.
- Усередині: `settings.py` — налаштування, `urls.py` — головний роутер, `wsgi/asgi.py` — точки входу для деплою.
- `startapp` генерує папку модуля з `models.py`, `views.py`, `admin.py`, `apps.py`, `tests.py`, `migrations/` (без `urls.py`).
- Структура «вмикається» через `settings.py` (`INSTALLED_APPS`, `ROOT_URLCONF`, `TEMPLATES`) — папка без запису в settings не працює.
- Шаблони у `templates/` кладуть у підпапки за іменем модуля, щоб уникнути конфліктів імен.
