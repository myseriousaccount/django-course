# JavaScript у Django: навіщо, коли і приклади

Рано чи пізно ти помічаєш: одного Django не вистачає — щось треба «оживити» на сторінці. Кнопка «подобається» під статтею блогу, модальне вікно з описом книги, жива пошук фільму під час набору. Цей урок пояснює, **де закінчується Django і починається JavaScript**, коли JS справді потрібен, а коли ні, і показує типові приклади з **різних доменів**.

## Чому Django-проєкту взагалі потрібен JS

> **Django працює на сервері, JavaScript — у браузері.** Django генерує готовий HTML і віддає його; JS виконується вже на боці користувача й змінює сторінку **без звернення до сервера**.

Це два різні світи, які **доповнюють** один одного, а не конкурують:

| | Django (сервер) | JavaScript (браузер) |
|---|---|---|
| Де виконується | на сервері, до відправлення сторінки | у браузері, коли сторінка вже відкрита |
| Що вміє | БД, логіка, рендер HTML, авторизація | реакція на кліки/введення, зміна сторінки на льоту |
| Чого НЕ вміє | змінити сторінку без перезавантаження | зайти в БД, зробити щось «секретне» безпечно |

> 🧠 Аналогія: Django — це **кухня**, що готує й видає готову страву (сторінку). JavaScript — це **офіціант у залі**: він реагує на клієнта тут і зараз (долив води, підсунув стілець), не бігаючи на кухню щоразу. Коли ж треба «нову страву» (дані з БД) — офіціант усе одно йде на кухню (робить запит до Django).

Ключова думка: усе, що потребує **миттєвої реакції в браузері без перезавантаження**, — це робота JS, а не Django.

## Коли JS потрібен, а коли — ні

Не хапайся за JS одразу: багато речей Django робить сам, простіше й надійніше.

**JS потрібен:**
- інтерактив без перезавантаження — модальні вікна, вкладки, акордеон (FAQ), бургер-меню;
- динамічні форми — додати/прибрати поле, залежні списки (країна → місто);
- **AJAX** — надіслати/отримати дані без перезавантаження (лайк/обране, жива пошук, безкінечний скрол);
- віджети — календар-датапікер, графіки, каруселі, карти;
- дрібниці UX — підтвердження перед видаленням, копіювання в буфер, таймер.

**JS НЕ потрібен (Django впорається сам):**
- звичайна навігація між сторінками → достатньо `<a href>`;
- звичайна форма → `POST` + `redirect` (урок про форми);
- просто показати дані з БД → шаблон рендерить сервер.

> 💡 Принцип **progressive enhancement**: спершу зроби, щоб працювало на чистому HTML/Django, а JS додавай зверху для зручності. Тоді сайт не «ламається», навіть якщо скрипт не завантажився.

## Як підключити JS у Django

JS-файл — це **статика**, тож усе за правилами уроку «Статика»:

```html
{% load static %}
<script src="{% static 'js/app.js' %}" defer></script>
```

- клади скрипти в `static/js/`;
- `defer` (або `<script>` перед `</body>`) — щоб JS виконувався, коли HTML уже побудовано;
- у `base.html` зручно мати `{% block scripts %}{% endblock %}`, щоб окремі сторінки додавали свій JS.

## Приклади — від простого до складного

### 1. Показати/сховати (без сервера) — меню, FAQ

Найпростіший випадок: JS лише перемикає клас, сервер не задіяний.

```html
<button onclick="document.getElementById('menu').classList.toggle('hidden')">☰ Меню</button>
<nav id="menu" class="hidden">...</nav>
```

### 2. Підтвердження перед видаленням

```html
<form method="post" action="{% url 'post_delete' post.id %}"
      onsubmit="return confirm('Точно видалити статтю?')">
    {% csrf_token %}
    <button>Видалити</button>
</form>
```

Якщо користувач натисне «Скасувати» — `confirm` поверне `false`, і форма не відправиться.

### 3. AJAX: дія без перезавантаження (fetch → Django → JsonResponse)

Ось найтиповіший «дорослий» випадок. Кнопка «Подобається» під статтею блогу шле запит до Django й оновлює себе, **не перезавантажуючи сторінку**.

**View у Django** повертає не HTML, а JSON:

```python
from django.http import JsonResponse

def toggle_like(request, post_id):
    post = get_object_or_404(Post, pk=post_id)
    # ... логіка додавання/зняття лайка ...
    return JsonResponse({'liked': True, 'count': post.likes.count()})
```

**JS у шаблоні** робить `fetch` і обробляє відповідь:

```html
<button id="like" data-url="{% url 'toggle_like' post.id %}">♡</button>

<script>
document.getElementById('like').addEventListener('click', async (e) => {
    const resp = await fetch(e.target.dataset.url, {
        method: 'POST',
        headers: {
            'X-CSRFToken': '{{ csrf_token }}',   // ← без цього Django відхилить (403)
        },
    });
    const data = await resp.json();
    e.target.textContent = data.liked ? `♥ ${data.count}` : `♡ ${data.count}`;
});
</script>
```

> ⚠️ **Найчастіша Django-пастка з AJAX — CSRF.** Django блокує будь-який `POST` без CSRF-токена (помилка **403**). Для `fetch` токен передають у заголовку `X-CSRFToken` (значення береш із `{{ csrf_token }}` або з cookie `csrftoken`). З `GET`-запитами цієї проблеми немає.

### 4. Жива пошук (fetch під час набору)

Той самий принцип: JS слухає введення в поле, шле `fetch` на Django-endpoint, той повертає JSON, а JS вставляє результат у сторінку. Приклад — пошук фільму:

```python
# cinema/views.py
def movie_search(request):
    q = request.GET.get('q', '')
    movies = Movie.objects.filter(title__icontains=q)[:10]
    return JsonResponse({
        'results': [{'id': m.id, 'title': m.title} for m in movies]
    })
```

```html
<input id="q" placeholder="Пошук фільму…">
<ul id="results"></ul>

<script>
document.getElementById('q').addEventListener('input', async (e) => {
    const resp = await fetch(`{% url 'movie_search' %}?q=${e.target.value}`);
    const data = await resp.json();
    document.getElementById('results').innerHTML =
        data.results.map(m => `<li>${m.title}</li>`).join('');
});
</script>
```

Тут `GET`-запит, тож CSRF не потрібен. Для великого трафіку варто ще додати **debounce** (не слати запит на кожну літеру, а з невеликою затримкою) — але це вже деталь оптимізації.

## Не писати все руками: htmx і Alpine.js

Часто JS потрібен для дрібниць — і писати щоразу `addEventListener` втомлює. У Django-світі популярні дві легкі бібліотеки:

- **htmx** — дуже «джанговий» підхід: AJAX задається **HTML-атрибутами**, а Django у відповідь віддає **шматочки HTML** (а не JSON). Майже без власного JS. Ідеально лягає на серверний рендер Django.
- **Alpine.js** — крихітна реактивність прямо в атрибутах (показати/сховати, лічильники, вкладки).

> 💡 До речі, ця навчальна платформа, яку ти зараз читаєш, зроблена саме на **Alpine.js** — уся інтерактивність (вкладки, тема, прогрес) — це атрибути `x-data`, `x-show` у HTML.

Великі фреймворки (**React**, **Vue**) для звичайного Django-сайту зазвичай **overkill** — їх беруть, коли будують окремий повноцінний SPA, а Django лишають як API.

## Типові помилки / Нюанси

> ⚠️ Забути CSRF-токен у `fetch`-POST → **403 Forbidden**. Це помилка №1 у новачків з AJAX у Django.

> ⚠️ Не клади в JS перевірки безпеки чи секрети — код у браузері **відкритий** кожному. Валідацію й права **завжди** дублюй/роби на сервері (у Django). JS — лише про зручність, не про захист.

> 💡 Скрипт із `defer` або перед `</body>` — інакше JS може виконатись раніше, ніж елемент з'явиться в DOM, і `getElementById` поверне `null`.

## Підсумок

- **Django — сервер (рендерить HTML), JS — браузер (інтерактив без перезавантаження).** Вони доповнюють одне одного.
- JS потрібен для дій «тут і зараз»: модалки, AJAX, віджети, підтвердження. Для звичайної навігації й форм — **ні**, це робота Django.
- Підключення: JS — це статика (`static/js/` + `{% static %}` + `defer`).
- **AJAX до Django:** `fetch` → view → `JsonResponse`; обов'язково передавай **CSRF-токен** у заголовку для `POST`.
- Щоб не писати гори JS — бери **htmx** (Django повертає HTML) або **Alpine.js** (реактивність в атрибутах); React/Vue — лише для повноцінних SPA.
- Безпека й валідація — **завжди на сервері**, ніколи не покладайся на JS.

> 📖 Деталі — у документації Django: «How to use Django's CSRF protection» (розділ про AJAX) та `JsonResponse` у «Request and response objects» (docs.djangoproject.com).
