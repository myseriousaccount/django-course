# Що таке Django

Перш ніж писати код, варто чітко зрозуміти, **чим Django відрізняється від звичного тобі Flask** і чому його структура виглядає саме так. Якщо коротко: **Flask дає тобі коробку з деталями, Django дає зібраний автомобіль**. Обидва — про створення вебдодатків на Python, але філософія в них протилежна.

## Визначення

> **Django** — це «full-stack» вебфреймворк мовою Python, побудований за принципом **«batteries included»** («батарейки в комплекті»). Тобто все, що типово потрібно сайту, вже всередині: ORM (робота з БД), система маршрутів, шаблони, форми, автентифікація і навіть готова **адмін-панель**.

Простими словами: Django дає не цеглу, а вже зведений каркас будинку. Твоє завдання — облаштувати його під конкретний проєкт, а не закладати фундамент щоразу з нуля.

## Flask vs Django — головна різниця

**Визначення.** Flask — це **мікрофреймворк** (дає мінімум, решту добираєш сам), Django — **full-stack** (дає майже все одразу, за конвенціями).

**Як це працює.** Найкоротше порівняння — у таблиці:

| | Flask (мікрофреймворк) | Django (full-stack) |
|---|---|---|
| Філософія | Дай мінімум, решту обери сам | Дай усе одразу, за конвенціями |
| ORM | Окремо (Flask-SQLAlchemy) | Вбудований (Django ORM) |
| Адмінка | Пишеш сам | Готова з коробки (`/admin`) |
| Структура | Довільна | **Жорстко задана** конвенціями |
| Міграції БД | Окремо (Flask-Migrate) | Вбудовані (`makemigrations`) |
| Автентифікація | Окремо (Flask-Login) | Вбудована (`django.contrib.auth`) |
| Форми й валідація | Окремо (WTForms) | Вбудовані (`django.forms`) |
| Роутинг | Декоратори (`@app.route`) | Окремі файли `urls.py` |

**Навіщо.** У Flask ти сама вирішувала, які папки створити, де лежатиме `models.py`, чи робити blueprint. У Django **більшість цих рішень уже прийнято за тебе** — є «правильний» спосіб, і фреймворк його очікує. Це менше свободи, але й менше питань «а як правильно».

## Патерн MTV (а не MVC)

**Визначення.** Класичний патерн вебдодатків — MVC (Model–View–Controller). Django використовує свій варіант — **MTV** (Model–Template–View).

**Як це працює.** Кожен шар відповідає за свою частину. Розберемо всі три детально — на прикладі уявного **блогу**:

| Шар | Що це | Приклад (блог) |
|---|---|---|
| **M** — Model | Опис даних і таблиць БД: класи Python, що описують структуру | `Post`, `Comment`, `Category` у `blog/models.py` |
| **T** — Template | HTML-шаблон, що бачить користувач; сюди підставляють дані | `blog/post_list.html` |
| **V** — View | Логіка: приймає запит, дістає дані з моделі, вибирає шаблон | `post_list()` у `blog/views.py` |

Розглянемо кожен шар окремо:

- **Model** — це «форма даних». Клас `Post` описує, що в поста є заголовок, текст, дата. Django сам перетворює цей клас на таблицю в БД (через ORM).
- **Template** — це «одяг» для даних: HTML з дірками `{{ }}`, куди підставляються значення.
- **View** — це «диспетчер»: приймає `request`, звертається до моделі (`Post.objects.all()`), кладе результат у шаблон і повертає готову сторінку.

⚠️ **Пастка в назвах.** Те, що в MVC називають «контролером», у Django називається **View**. А «View» з MVC (відображення) у Django — це **Template**. Тобто:

```
Запит → URL-маршрут → View (логіка) → Model (дані) + Template (HTML) → Відповідь
```

**Приклад руху запиту.** Коли користувач відкриває сторінку зі списком статей блогу:

1. Django дивиться у `root/urls.py`, потім у `blog/urls.py` — який маршрут підходить під адресу.
2. Знаходить, що `''` веде до view `post_list`.
3. `post_list` у `blog/views.py` дістає дані через модель: `Post.objects.all()`.
4. Ця ж view викликає `render(...)` з шаблоном `blog/post_list.html`.
5. Готовий HTML повертається в браузер.

> 🧠 Аналогія: **View — це офіціант**. Він приймає замовлення (`request`), іде на кухню по страву (Model), викладає її на гарну тарілку (Template) і виносить клієнту (Response). Сам не готує і не миє посуд — лише координує.

## «Batteries included» — повний склад «батарейок»

**Визначення.** «Батарейки» — це вбудовані застосунки Django, які дають готову функціональність без сторонніх бібліотек. Вони лежать у пакеті `django.contrib`.

**Як це працює.** Те, що в Flask ти збирала з кількох бібліотек, у Django вже прописане в `INSTALLED_APPS` (глянь свій `root/settings.py`):

```python
INSTALLED_APPS = [
    'django.contrib.admin',        # готова адмін-панель на /admin
    'django.contrib.auth',         # користувачі, групи, права, логін, паролі
    'django.contrib.contenttypes', # облік типів моделей (для прав і зв'язків)
    'django.contrib.sessions',     # сесії (кошик, «запам'ятати мене»)
    'django.contrib.messages',     # flash-повідомлення («Збережено!»)
    'django.contrib.staticfiles',  # роздача CSS/JS/картинок
    'blog',                        # ← твої власні модулі (приклад)
    'users',
    'comments',
]
```

Перші шість рядків — це «батарейки», які Django поклав сам одразу після `startproject`. Далі йдуть модулі (apps), які створюєш ти.

Окрім тих, що ввімкнені за замовчуванням, у `django.contrib` є ще й інші «батарейки», які підключають за потреби:

| «Батарейка» | Що дає |
|---|---|
| `django.contrib.admin` | Адмін-панель для керування даними |
| `django.contrib.auth` | Автентифікація: користувачі, логін, паролі, права |
| `django.contrib.sessions` | Серверні сесії між запитами |
| `django.contrib.messages` | Одноразові повідомлення користувачу |
| `django.contrib.staticfiles` | Керування статичними файлами |
| `django.contrib.humanize` | Гарне форматування («2 хвилини тому», «1 500» → «1 500») |
| `django.contrib.sitemaps` | Генерація `sitemap.xml` для SEO |
| `django.contrib.syndication` | RSS/Atom-стрічки |
| `django.contrib.gis` | Гео-дані (карти, координати) |

**Навіщо.** У Flask типовий сайт із користувачами = Flask + Flask-SQLAlchemy + Flask-Migrate + Flask-Login + WTForms + окремо написана адмінка. У Django все це — одна установка й кілька рядків у `settings.py`.

## Коли обирати Django, а коли Flask

**Навіщо.** Жоден фреймворк не «кращий» — вони для різних задач. Орієнтир простий:

- **Django** — коли є БД, користувачі, адмінка, форми: типовий «сайт із вмістом». Приклади: блог, інтернет-магазин, бібліотечний каталог, кіно-довідник, соцмережа. Менше рішень — швидший старт.
- **Flask** — коли треба щось маленьке й нестандартне: мікросервіс, API на пару ендпоінтів, телеграм-бот, або коли хочеш повний контроль над структурою.

🧠 Хороший маркер: якщо в проєкті буде **адмінка для контент-менеджера** і **таблиці, пов'язані між собою** (пости й коментарі, товари й замовлення, книги й автори) — це кейс для Django. «Батарейки» там економлять найбільше часу.

## Типові помилки / Нюанси

> ⚠️ **Плутанина MVC ↔ MTV.** Не шукай у Django «контролер» — його роль виконує **View**. А те, що в MVC зветься View, тут — **Template**.

> ⚠️ **«Django = тільки великі сайти».** Ні. Django чудово підходить і для маленького проєкту з БД — просто ти не використаєш усі «батарейки». Розмір проєкту не аргумент проти.

> 💡 **App ≠ проєкт.** Django-проєкт складається з кількох app-ів (модулів). Один app = одна функціональна область (блог, користувачі, коментарі). Про це — окремий розділ.

## Підсумок

- **Django** — full-stack фреймворк за принципом «batteries included»: ORM, адмінка, auth, міграції, форми, сесії — вже всередині.
- Патерн — **MTV**: Model (дані), Template (HTML), View (логіка-диспетчер). «View» у Django = «контролер» у MVC, а «Template» = «view» у MVC.
- «Батарейки» лежать у `django.contrib`; шість із них увімкнені одразу, решта — за потреби (`humanize`, `sitemaps`, `gis`…).
- На відміну від Flask, **структура задана конвенціями** — є очікуваний «правильний» спосіб, менше рішень щоразу.
- Django обирають, коли є БД, користувачі й адмінка (блог, магазин, каталог); Flask — для малого й нестандартного.

> 📖 Докладніше про загальний огляд фреймворку — у розділі «Django at a glance» офіційної документації Django (docs.djangoproject.com).
