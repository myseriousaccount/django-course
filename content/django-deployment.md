# Деплой: викласти Django на прод

Деплой — це коли твій проєкт із локального комп'ютера перебирається на справжній сервер в інтернеті, де його бачать реальні люди. Цей урок пояснює, чим прод відрізняється від розробки, який чекліст пройти перед виходом, як роздавати статику, який сервер запускати замість `runserver` і де хостити. Приклади з різних доменів, щоб схема була універсальна.

## dev vs prod коротко

> **dev** (development) — режим розробки на твоєму комп'ютері: зручність і докладні помилки. **prod** (production) — «бойовий» режим на сервері: швидкість і безпека для реальних користувачів.

**Як це працює.** Той самий код в цих режимах поводиться по-різному — керує цим передусім налаштування `DEBUG`. У dev воно `True` (показує детальні сторінки помилок), у prod — `False` (нікому не видно нутрощів).

**Навіщо.** Прод бачать сторонні люди, зокрема зловмисники. Те, що на dev допомагає (докладний трейсбек, автоперезапуск, вбудований сервер), на проді стає діркою в безпеці або гальмом. Про розділення налаштувань і секрети — окремий урок «dev vs prod і секрети»; тут — практика виходу на прод.

> <i class="bi bi-lightbulb"></i> Уяви кухню ресторану. Dev — це домашня кухня, де ти пробуєш рецепт: усе під рукою, помилки не страшні. Prod — це кухня для відвідувачів: тут інші вимоги до чистоти, швидкості й безпеки, бо їдять реальні гості.

## Чекліст перед продом

Перш ніж викладати проєкт, пройди список обов'язкових пунктів:

| Пункт | Було на dev | Має бути на prod |
|---|---|---|
| `DEBUG` | `True` | **`False`** |
| `ALLOWED_HOSTS` | `[]` або `['*']` | справжні домени: `['shop.example.com']` |
| `SECRET_KEY` | у коді | з **env-змінної**, не в репозиторії |
| База даних | SQLite | справжня, зазвичай **PostgreSQL** |

**Як це працює.** З `DEBUG=False` Django перестає показувати трейсбеки й **вимагає** заповнений `ALLOWED_HOSTS` — список доменів, з яких дозволено приймати запити. Порожній список при вимкненому `DEBUG` = сайт відмовиться відповідати.

**Навіщо.** Кожен пункт закриває ризик: `DEBUG=False` ховає нутрощі, `ALLOWED_HOSTS` відсікає чужі домени, `SECRET_KEY` з env не потрапляє в Git, PostgreSQL витримує реальне навантаження (SQLite — це файл, він не для багатьох одночасних записів).

Django вміє перевірити готовність сам:

```bash
python manage.py check --deploy
```

Ця команда проходить по налаштуваннях безпеки й показує попередження: слабкий `SECRET_KEY`, забутий HTTPS, увімкнений `DEBUG` тощо.

> <i class="bi bi-info-circle"></i> Запусти `check --deploy` **до** першого деплою й виправ усі попередження. Це найдешевший спосіб знайти діри в безпеці — Django сам їх перелічить.

## Статика на проді

> **Статика** — це файли, що не змінюються від запиту до запиту: CSS, JavaScript, зображення інтерфейсу.

**Як це працює.** У dev статику роздавав сам Django (`runserver`). На проді він цього **не робить** — це неефективно. Спершу всю статику з усіх застосунків збирають в одну папку:

```python
# settings.py
STATIC_URL = '/static/'
STATIC_ROOT = BASE_DIR / 'staticfiles'    # куди зібрати
```

```bash
python manage.py collectstatic           # зібрати все в STATIC_ROOT
```

Потім цю папку роздає щось швидке — **WhiteNoise** (додаток до самого Django, найпростіше для новачка) або **Nginx** (окремий веб-сервер попереду).

**Навіщо.** `STATIC_ROOT` — це «склад» зібраної статики; `collectstatic` наповнює його перед кожним деплоєм. Роздавати статику має спеціалізований інструмент, а не Python-код — так набагато швидше.

> <i class="bi bi-pin-angle"></i> **Media** (файли, які завантажують користувачі: аватарки, фото товарів) — це **не статика**. Вона живе окремо (`MEDIA_ROOT`), `collectstatic` її не чіпає, і роздають її теж окремо.

## Сервер застосунку

> **`runserver` — НЕ для прода.** Це навчальний однопотоковий сервер: зручний, але повільний і небезпечний під навантаженням.

**Як це працює.** На проді твій код запускає **сервер застосунку** — програма, що спілкується з Django за стандартом **WSGI** (синхронний) або **ASGI** (асинхронний). Найпоширеніший вибір — **Gunicorn**:

```bash
gunicorn root.wsgi        # root — папка з settings.py та wsgi.py
```

Тут `root.wsgi` вказує на файл `wsgi.py`, який Django створив у папці проєкту разом із `settings.py`.

**Навіщо.** Gunicorn запускає кілька робочих процесів (workers), тому обслуговує багато користувачів одночасно й не падає від навантаження. Альтернативи: **uWSGI** (теж WSGI) або **Uvicorn** (для ASGI, якщо в проєкті є асинхронні view).

> <i class="bi bi-info-circle"></i> Правило просте: `runserver` — тільки поки ти розробляєш локально. Щойно проєкт на сервері для людей — Gunicorn або аналог.

## Як усе складається разом

Класична схема прод-сервера — три ланки:

```
                    ┌──────────────────────────────┐
   користувач  →    │  Nginx                        │
   (браузер)        │  • віддає статику й media     │
                    │  • проксіює решту далі  ──┐   │
                    └──────────────────────────┼───┘
                                               ↓
                                     ┌───────────────────┐
                                     │  Gunicorn         │
                                     │  (сервер WSGI)    │
                                     └─────────┬─────────┘
                                               ↓
                                     ┌───────────────────┐
                                     │  Django (твій код)│
                                     │      ↕            │
                                     │  PostgreSQL       │
                                     └───────────────────┘
```

**Як це працює.** **Nginx** стоїть попереду: сам віддає статику й media (це він робить блискавично), а всі «живі» запити передає **Gunicorn**, який викликає твій **Django**, а той ходить у **PostgreSQL**. Кожен займається своїм.

**Навіщо.** Розподіл праці: Nginx — швидка роздача файлів і захист, Gunicorn — запуск Python, Django — логіка. Разом це витримує реальний трафік.

## Де хостити

Є три поширені варіанти — від найпростішого до найгнучкішого:

- **PaaS** (Render, Railway, Fly.io) — платформа сама піднімає сервер, БД, HTTPS. Ти лише під'єднуєш репозиторій. **Найпростіше для новачка.**
- **VPS** (звичайний віртуальний сервер) — повний контроль, але Nginx, Gunicorn, БД і HTTPS налаштовуєш руками.
- **Docker** — пакуєш застосунок у контейнер, який однаково запускається будь-де. Зручно, коли проєкт росте.

> <i class="bi bi-lightbulb"></i> Порада новачку: почни з **PaaS**. Render чи Railway приховують більшість складнощів — розберешся з деплоєм концептуально, не потонувши в налаштуванні Nginx. VPS і Docker освоїш пізніше.

## Кроки на сервері

Хоч який хостинг, послідовність виходу на прод майже однакова:

```bash
pip install -r requirements.txt    # 1. поставити залежності
# 2. задати env-змінні: SECRET_KEY, DEBUG=False, DATABASE_URL, ALLOWED_HOSTS
python manage.py migrate            # 3. створити таблиці в проді
python manage.py collectstatic      # 4. зібрати статику
gunicorn root.wsgi                  # 5. запустити сервер застосунку
```

**Навіщо.** Порядок важливий: спершу код і залежності, потім секрети через оточення, далі БД і статика, і аж наприкінці — запуск. Пропустиш `migrate` — сайт впаде на відсутній таблиці; пропустиш `collectstatic` — сторінки будуть без стилів.

## Типові помилки / Нюанси

> <i class="bi bi-exclamation-triangle"></i> **`DisallowedHost` після деплою** — забула додати домен у `ALLOWED_HOSTS`. З `DEBUG=False` він обов'язковий і мусить містити реальну адресу сайту.

> <i class="bi bi-exclamation-triangle"></i> **Сайт без стилів** — не запущено `collectstatic` або не налаштовано роздачу статики (WhiteNoise/Nginx). У проді Django сам CSS не віддає.

> <i class="bi bi-exclamation-triangle"></i> **Секрети в Git** — `SECRET_KEY`, паролі БД, файл SQLite. Ніколи не коміт їх: додай у **`.gitignore`**, а значення передавай через env-змінні.

> <i class="bi bi-info-circle"></i> Увімкни **HTTPS** — без нього паролі й куки летять відкрито. PaaS зазвичай дає сертифікат автоматично; на VPS ставлять безкоштовний Let's Encrypt.

> <i class="bi bi-pin-angle"></i> Налаштуй **логи**. З `DEBUG=False` користувач бачить лише «щось пішло не так» — деталі помилки шукатимеш саме в логах сервера.

## Підсумок

- **Деплой** — переїзд проєкту на сервер; прод відрізняється від dev режимом безпеки й швидкості.
- **Чекліст:** `DEBUG=False`, заповнений `ALLOWED_HOSTS`, `SECRET_KEY` з env, PostgreSQL замість SQLite; перевір усе через `manage.py check --deploy`.
- **Статика:** `STATIC_ROOT` + `collectstatic`, роздача через **WhiteNoise** або **Nginx**; media — окремо.
- **Сервер:** `runserver` не для прода → **Gunicorn** (`gunicorn root.wsgi`) чи uWSGI/Uvicorn.
- **Схема:** Nginx (статика + проксі) → Gunicorn → Django → PostgreSQL.
- **Хостинг:** PaaS (найпростіше — почни звідси), VPS, Docker.
- **Не комітити** секрети й базу (`.gitignore`), увімкнути HTTPS, налаштувати логи.
