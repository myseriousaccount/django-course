# Модулі (apps): кожна частина = application

У Django проєкт не пишуть як один суцільний файл — його **ділять на модулі, які називаються apps**. Цей урок пояснює, що таке app, навіщо саме така структура, як його створити й зареєструвати. Структуру shop-app беремо як живий приклад, а самі «теми» apps показуємо з різних доменів, щоб ти бачила: поділ універсальний.

## Що таке app

> **App** — це самодостатня частина проєкту, що відповідає за **одну функціональну область**. Один app = одна «тема»: товари, статті блогу, книги в бібліотеці, користувачі.

**Як це працює.** Кожен app збирає в собі все, що стосується його теми: свої моделі, view, маршрути, шаблони, реєстрацію в адмінці. Ось як одна й та сама ідея розкладається на різних проєктах:

| Проєкт | Приклади apps | За що відповідає кожен |
|---|---|---|
| shop-app | `home`, `catalog`, `carts`, `accounts`, `orders` | головна, каталог товарів, кошик, користувачі, замовлення |
| блог | `blog`, `comments`, `accounts` | статті, коментарі, автори |
| бібліотека | `catalog`, `loans`, `members` | книги, видача, читачі |

> <i class="bi bi-lightbulb"></i> Уточнення формулювання «кожна сторінка = application». Точніше: **кожна функціональна область = app**, а не буквально кожна сторінка. У модуля `home` три сторінки (index, about, contacts) — і всі вони в одному app, бо логічно пов'язані. У блозі сторінки «список статей» і «одна стаття» теж живуть в одному app `blog`. Дроби на apps **за змістом**, а не за кількістю сторінок.

## Навіщо ділити на apps

**Проблема, яку це розв'язує.** Якщо весь код проєкту лежить в одному місці, він швидко перетворюється на хаос: важко орієнтуватися, важко змінювати одну частину, не зачепивши інших. Поділ на apps дає три практичні переваги:

1. **Порядок.** Код статей — у `blog`, код коментарів — у `comments`, код книг — у `catalog`. Не треба шукати все в одному гігантському файлі.
2. **Незалежність.** Кожен app має власні `models`, `views`, `urls`. Можна працювати над коментарями, не зачіпаючи статті.
3. **Перевикористання.** Добре зроблений app можна перенести в інший проєкт майже без змін. App `accounts` (реєстрація/логін) однаково потрібен і магазину, і блогу, і бібліотеці.

> <i class="bi bi-info-circle"></i> Якщо згадати Flask — це дуже схоже на **blueprints**. Там ти ділила Clinic-app на `auth/`, `admin/`, `core/`. App у Django — це та сама ідея модульності, тільки «офіційна» й вбудована у фреймворк.

## Як створити app

App створюють однією командою через `manage.py`:

```bash
python manage.py startapp blog
```

Django згенерує папку `blog/` з готовим набором файлів:

```
blog/
├── __init__.py
├── admin.py
├── apps.py
├── models.py
├── tests.py
├── views.py
└── migrations/
    └── __init__.py
```

<i class="bi bi-exclamation-triangle"></i> Зверни увагу: `startapp` **не створює** `urls.py` — цей файл ти додаєш сама, коли модулю потрібні власні маршрути. Про призначення кожного з цих файлів — у наступному уроці «Анатомія app».

### Створити кілька apps одразу

`startapp` приймає **лише одне ім'я** — передати кілька в одній команді не можна. Але через термінал є три способи зробити це швидко:

**1. По черзі** — просто кілька команд підряд:

```bash
python manage.py startapp blog
python manage.py startapp comments
python manage.py startapp accounts
```

**2. Ланцюжком через `&&`** — в один рядок (кожна наступна виконається, лише якщо попередня вдалася):

```bash
python manage.py startapp blog && python manage.py startapp comments && python manage.py startapp accounts
```

**3. Циклом** (bash/zsh) — найзручніше для багатьох одразу. Так можна створити **всі** модулі проєкту однією дією:

```bash
for app in home catalog carts accounts orders; do
    python manage.py startapp $app
done
```

> <i class="bi bi-exclamation-triangle"></i> Хоч би скільки apps ти створила за раз, **кожен** усе одно доведеться окремо додати в `INSTALLED_APPS` (про це — наступний розділ). Термінал створює папки, але реєстрацію за тебе не зробить.

## Крок, який легко забути: реєстрація

**Як це працює.** Створити app мало — Django про нього ще не знає. Треба **увімкнути** його у `settings.py`:

```python
# root/settings.py
INSTALLED_APPS = [
    'django.contrib.admin',
    # ... вбудовані «батарейки» ...
    'blog',        # ← ось так app стає "видимим"
    'comments',
    'accounts',
]
```

Без цього запису:
- моделі модуля не потраплять у міграції,
- шаблони модуля не знайдуться,
- адмінка його проігнорує.

### Типова помилка

> <i class="bi bi-exclamation-octagon"></i> Класична ситуація новачка: `startapp blog`, написала модель `Post`, запустила `makemigrations` — а Django каже «no changes». Причина майже завжди — **забули додати app у `INSTALLED_APPS`**.

## Проєкт і apps разом

```
root/ (проєкт)
│   settings.py  — список усіх apps (INSTALLED_APPS)
│   urls.py      — делегує маршрути модулям через include()
│
├── blog/      (app)  — свої views, urls, models (статті)
├── comments/  (app)  — свої views, urls, models (коментарі)
└── accounts/  (app)  — реєстрація, логін
```

Розподіл ролей простий: **проєкт — це «диспетчер»**, він знає про всі apps і направляє запити. **Apps — це «виконавці»**: кожен робить свою частину роботи. Та сама схема діє в shop-app (`catalog`, `carts`…), у бібліотеці, будь-де — змінюються лише назви й теми.

## Підсумок

- **App** — самодостатня частина проєкту під одну функціональну область (статті, книги, товари, користувачі…).
- Точніше за «сторінка = app»: **функціональна область = app**; кілька пов'язаних сторінок живуть в одному app.
- Створення: `python manage.py startapp <ім'я>` (але `urls.py` додаєш сама); кілька apps — по черзі, через `&&` або циклом.
- App обов'язково треба **зареєструвати** в `INSTALLED_APPS`, інакше Django його не бачить (типова причина «no changes» у міграціях).
- Концептуально app = Flask-blueprint, тільки вбудований у фреймворк.
