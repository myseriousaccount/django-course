# Валідація: клієнт проти сервера

Коли ти будуєш форму — логін, реєстрацію, публікацію статті, замовлення, рецензію, контактну форму — виникає природне питання: де саме перевіряти введені дані? У браузері скриптом чи вже на сервері у Django? Цей урок дає чітку відповідь і показує, як валідувати **будь-яке** введення користувача, а не лише авторизацію. Приклади навмисно з **різних доменів** (блог, магазин, кінотека, бібліотека), бо форми в проєкті є всюди — і кожна з них має захищати твою базу.

## Дві лінії валідації

**Визначення.** Валідація буває **клієнтська** і **серверна** — це дві різні лінії оборони, що працюють у різних місцях.

- **Клієнтська** відбувається у браузері користувача: атрибути HTML5 (`required`, `type="email"`, `minlength`) і, за бажанням, JavaScript.
- **Серверна** відбувається у Django вже після того, як дані прийшли на сервер: `Form` / `ModelForm` та логіка у view.

**Як це працює.** Уяви прикордонний контроль 🧠. Клієнтська перевірка — це доброзичливий консультант ще в черзі: він підказує «у тебе не заповнене поле» до того, як ти дійшов до віконця. Серверна перевірка — це сам прикордонник за склом: саме він приймає остаточне рішення. Консультанта можна оминути (просто пройти повз), а прикордонника — ні.

**Навіщо.** Дві лінії служать різним цілям: клієнтська дбає про **зручність**, серверна — про **коректність і безпеку**. Плутати їхні ролі — головна помилка новачків.

## Золоте правило

> ⚠️ **Серверна валідація — обов'язкова ЗАВЖДИ. Клієнтська — лише приємний UX і НЕ замінює серверну.** Ніколи не довіряй даним, що прийшли з браузера.

**Як це працює.** Проблема клієнтської валідації в тому, що вона виконується **на чужій території** — на комп'ютері користувача, який ти не контролюєш. JS можна вимкнути в один клік. HTML5-атрибути можна стерти прямо в DevTools. А головне — запит можна взагалі **надіслати в обхід сторінки**: через `curl`, Postman чи ту саму панель розробника — POST напряму на твій URL, минаючи будь-яку JS-перевірку.

```bash
# Ось так дані летять на сервер повз усі твої браузерні перевірки:
curl -X POST https://твій-сайт/login/ -d "username=&password="
```

Сервер отримає порожні поля, ніби ніякого `required` і не існувало. Тому:

> 💡 Клієнтська перевірка нічого не **гарантує** серверу — вона лише *ймовірно* фільтрує чесних користувачів. Зловмиснику вона не перешкода взагалі.

**Навіщо.** Якщо покластися лише на клієнта, у тебе відкрита діра: до бази потраплять будь-які дані, які надішле хто завгодно. Саме сервер — єдине місце, де перевірку **неможливо обійти**.

> 💡 Це стосується **кожної** форми, а не лише логіна. Форма публікації статті, оформлення замовлення, поле пошуку книги, рецензія на фільм — усе це введення користувача, і все проходить через сервер. Правило одне: *будь-яке введення — вороже, поки сервер не довів протилежне.*

## Пряма відповідь: чи потрібна JS-валідація для логіна?

**Визначення.** JS-валідація для форми логіна — **не обов'язкова**.

**Як це працює.** Для коректного й безпечного логіна цілком достатньо серверної частини: Django-форма перевіряє, що поля заповнені, а функція `authenticate()` — що пара «логін + пароль» справді існує. JS тут нічого не додає до **правильності**: він лише дасть миттєвий фідбек («заповни пароль») без перезавантаження сторінки.

**Навіщо.** JS для логіна — це косметика заради UX, а не вимога. Хочеш плавніший досвід — додай; не хочеш — форма й без нього працюватиме коректно та безпечно.

## Пряма відповідь: чи повторювати перевірки у view, якщо вже є JS?

**Визначення.** **Так — сервер мусить перевіряти сам, навіть якщо JS уже все перевірив.**

**Як це працює.** Це не «зайве дублювання», а розподіл ролей: **клієнт = зручність, сервер = істина**. Але «перевірка на сервері» не означає ручні `if` у view. На практиці це означає — використати **Django-форму** та її метод `is_valid()`:

```python
from django.contrib.auth.forms import AuthenticationForm

def login_view(request):
    if request.method == "POST":
        form = AuthenticationForm(request, data=request.POST)
        if form.is_valid():                  # ← ось тут і є серверна валідація
            user = form.get_user()
            login(request, user)
            return redirect("home")
    else:
        form = AuthenticationForm()
    return render(request, "login.html", {"form": form})
```

Форма сама перевіряє заповненість полів і коректність облікових даних. Тобі не треба писати ці правила руками.

> 💡 Django-форма — це і є твоя «серверна перевірка». Один виклик `is_valid()` замінює десяток ручних `if`. Це прямий прояв **DRY**: правила описані в одному місці — у формі.

**Навіщо.** Якщо винести правила у форму, вони живуть в **одному джерелі**. JS може дублювати їх для UX, але істину визначає форма — і синхронізувати логіку не доводиться.

## Три рівні валідації в Django

**Визначення.** У типовому Django-проєкті дані проходять до трьох рівнів перевірки — від найдешевшого клієнтського до головного серверного.

**Як це працює.**

1. **HTML5** — `required`, `type="email"`, `minlength`, `maxlength`, `min`, `max`. Найдешевший рівень: браузер не пустить очевидно неправильну форму без жодного рядка коду.

   ```html
   <input type="email" name="email" required minlength="3">
   <input type="number" name="quantity" min="1" max="99" required>
   ```

2. **JavaScript (опційно)** — миттєвий фідбек без відправлення форми: підсвітити поле, показати «паролі не збігаються» одразу. Потрібен лише для UX.

3. **Django `Form` / `ModelForm`** — **основний** рівень. Сюди входять `is_valid()`, валідатори полів (`validators=[...]`), метод `clean_<field>()` для власних правил, метод `clean()` для міжпольових перевірок, а для паролів — `AUTH_PASSWORD_VALIDATORS` у налаштуваннях. Саме цей рівень — істина.

> 📌 Якщо приходиш із Flask: рівень 3 — це аналог **WTForms** (`form.validate_on_submit()`, валідатори `DataRequired`, `Length`, `NumberRange`). Django-форма грає ту саму роль єдиного джерела правил валідації, лише вбудована у фреймворк.

**Навіщо.** Рівні доповнюють один одного: HTML5 і JS покращують досвід чесного користувача, а Django-форма гарантує, що жоден запит не пройде повз правила — незалежно від браузера.

## Рівні валідації в Django-формі детально

Тепер розберемо, з чого саме складається той «третій рівень» — серверна форма. У Django є **чотири механізми**, і вони спрацьовують у чіткому порядку.

### 1. Валідатори полів (`validators=[...]`)

**Визначення.** Валідатор — це готова функція-перевірка, яку ти чіпляєш прямо до поля. Django має набір вбудованих валідаторів у `django.core.validators`.

**Як це працює.** Найпоширеніші:

| Валідатор | Що перевіряє |
|---|---|
| `MinValueValidator(x)` / `MaxValueValidator(x)` | число не менше / не більше за `x` |
| `MinLengthValidator(n)` / `MaxLengthValidator(n)` | довжину тексту |
| `RegexValidator(pattern)` | відповідність регулярному виразу |
| `EmailValidator()` | коректність email |

```python
from django import forms
from django.core.validators import MinValueValidator, RegexValidator

class BookForm(forms.Form):
    title = forms.CharField(
        max_length=200,
        validators=[
            RegexValidator(
                r"^[\w\s\-.,:]+$",
                message="Назва містить недопустимі символи.",
            )
        ],
    )
    isbn = forms.CharField(
        max_length=13,
        validators=[
            RegexValidator(r"^\d{13}$", message="ISBN має складатися з 13 цифр.")
        ],
    )
    published_year = forms.IntegerField(
        validators=[MinValueValidator(1450, message="Рік видання виглядає нереальним.")],
    )
```

**Навіщо.** Валідатор — це перевірка «одним рядком», яку легко перевикористати на багатьох полях. Не пиши те, що вже є у фреймворку.

> 💡 Це прямий аналог валідаторів WTForms (`NumberRange`, `Length`, `Regexp`). Ідея та сама: декларативний список перевірок біля поля.

### 2. `clean_<field>()` — валідація одного поля

**Визначення.** Метод `clean_<назва_поля>()` описує **власне** правило для конкретного поля — те, чого немає серед готових валідаторів (звернення до бази, заборонені слова, складна логіка).

**Як це працює.** Django викликає його автоматично під час `is_valid()`. Ти читаєш значення з `self.cleaned_data`, перевіряєш і **обовʼязково повертаєш** його назад:

```python
class PostForm(forms.Form):
    title = forms.CharField(max_length=200)

    FORBIDDEN = {"тест", "asdf", "123"}

    def clean_title(self):
        title = self.cleaned_data["title"].strip()
        if title.lower() in self.FORBIDDEN:
            raise forms.ValidationError("Це не схоже на справжній заголовок статті.")
        if len(title) < 5:
            raise forms.ValidationError("Заголовок має містити щонайменше 5 символів.")
        return title
```

**Навіщо.** Один метод — одне поле, одне правило. Помилка, кинута через `raise forms.ValidationError(...)`, автоматично прикріпиться саме до цього поля й покажеться користувачу поряд із ним.

> ⚠️ Не забувай `return` наприкінці `clean_<field>()`. Якщо нічого не повернути, поле стане `None` — це поширена помилка новачків.

### 3. `clean()` — міжпольова (крос-полева) валідація

**Визначення.** Метод `clean()` перевіряє **звʼязки між кількома полями** одразу — коли правильність одного поля залежить від іншого.

**Як це працює.** Класичні приклади: пароль має збігатися з підтвердженням; дата початку сеансу — бути раніше за дату завершення. Тут `clean_<field>()` не підходить, бо він бачить лише одне поле. Використовуй `clean()` і кидай помилку або на конкретне поле (`add_error`), або на форму загалом:

```python
class ScreeningForm(forms.Form):        # сеанс у кінотеатрі
    starts_at = forms.DateTimeField()
    ends_at = forms.DateTimeField()

    def clean(self):
        cleaned = super().clean()
        start = cleaned.get("starts_at")
        end = cleaned.get("ends_at")
        if start and end and start >= end:
            self.add_error("ends_at", "Кінець сеансу має бути пізніше за початок.")
        return cleaned
```

А ось той самий механізм для пари «пароль + підтвердження» у реєстрації:

```python
def clean(self):
    cleaned = super().clean()
    p1 = cleaned.get("password")
    p2 = cleaned.get("password_confirm")
    if p1 and p2 and p1 != p2:
        raise forms.ValidationError("Паролі не збігаються.")
    return cleaned
```

**Навіщо.** `clean()` — єдине місце, де видно всю форму цілком. Саме сюди виноситься логіка «поле A має узгоджуватися з полем B».

> 💡 Порядок такий: спершу валідатори полів → потім усі `clean_<field>()` → наприкінці один `clean()`. У `clean()` потрапляють лише ті поля, що вже пройшли свою індивідуальну перевірку.

### 4. Валідація файлів і зображень (коротко)

**Визначення.** Завантажені файли — теж введення користувача, і його теж треба перевіряти: розмір, розширення, тип.

**Як це працює.** Для розширень є готовий `FileExtensionValidator`; розмір перевіряєш у `clean_<field>()`:

```python
from django.core.validators import FileExtensionValidator

class CoverForm(forms.Form):            # обкладинка книги/фільму
    cover = forms.ImageField(
        validators=[FileExtensionValidator(["jpg", "jpeg", "png", "webp"])]
    )

    def clean_cover(self):
        img = self.cleaned_data["cover"]
        if img.size > 2 * 1024 * 1024:            # 2 МБ
            raise forms.ValidationError("Файл завеликий (максимум 2 МБ).")
        return img
```

**Навіщо.** Без перевірки хтось завантажить 500-мегабайтний файл чи виконуваний скрипт замість обкладинки. Розмір і розширення — мінімальний захист будь-якого поля з файлом.

## Приклади: форми є всюди

Валідація потрібна не лише для авторизації. Пройдімося реальними формами з **різних доменів** — кожна показує свій механізм.

### Публікація статті (`ModelForm`)

`ModelForm` бере поля з моделі, тож частина правил (типи, `max_length`) успадковується автоматично. Додаєш лише те, чого моделі бракує:

```python
from django import forms
from blog.models import Post

class PostForm(forms.ModelForm):
    class Meta:
        model = Post
        fields = ["title", "slug", "body"]

    def clean_body(self):
        body = self.cleaned_data["body"].strip()
        if len(body) < 50:
            raise forms.ValidationError("Текст статті замалий — напиши хоча б абзац.")
        return body

    def clean_slug(self):
        slug = self.cleaned_data["slug"]
        qs = Post.objects.filter(slug=slug).exclude(pk=self.instance.pk)
        if qs.exists():
            raise forms.ValidationError("Стаття з таким slug уже існує.")
        return slug
```

Тут працюють одразу три речі: успадкована з моделі перевірка довжини `title`, власна перевірка обсягу `body` та унікальності `slug`.

### Контактна форма (`forms.Form`)

Проста форма без моделі — бо їй нема що зберігати в БД, вона лише надсилає лист:

```python
class ContactForm(forms.Form):
    name = forms.CharField(max_length=80)
    email = forms.EmailField()                      # EmailField сам перевіряє формат
    message = forms.CharField(widget=forms.Textarea, min_length=10)

    def clean_message(self):
        msg = self.cleaned_data["message"].strip()
        if "http://" in msg or "https://" in msg:
            raise forms.ValidationError("Посилання у повідомленні заборонені (спам).")
        return msg
```

> 💡 `forms.Form` беруть, коли даним не потрібна модель (контакт, пошук, фільтр). `forms.ModelForm` — коли форма напряму створює/редагує обʼєкт у БД (стаття, книга, профіль). Правило вибору просте: *є модель для збереження — бери `ModelForm`.*

### Оформлення замовлення (`forms.Form`, кількість)

```python
class OrderItemForm(forms.Form):
    product_id = forms.IntegerField(widget=forms.HiddenInput)
    quantity = forms.IntegerField(min_value=1)

    def clean_quantity(self):
        quantity = self.cleaned_data["quantity"]
        if quantity > 99:
            raise forms.ValidationError("За раз можна замовити не більше 99 штук.")
        return quantity
```

> 💡 `min_value=1` у полі вже вбудовує `MinValueValidator` — тобі не треба писати його руками. Власний `clean_quantity` додаємо лише для правила, якого немає «з коробки» (верхня межа з бізнес-причин).

### Редагування профілю (унікальність, крім себе)

Тонкий, але дуже частий випадок. При редагуванні email має лишатися унікальним — але **не конфліктувати із самим користувачем**, якого ти зараз редагуєш. Тому в запиті виключаємо поточний обʼєкт через `.exclude(pk=...)`:

```python
from django.contrib.auth import get_user_model

User = get_user_model()

class ProfileForm(forms.ModelForm):
    class Meta:
        model = User
        fields = ["username", "email"]

    def clean_email(self):
        email = self.cleaned_data["email"]
        qs = User.objects.filter(email=email).exclude(pk=self.instance.pk)
        if qs.exists():
            raise forms.ValidationError("Ця пошта вже зайнята іншим користувачем.")
        return email
```

> ⚠️ Без `.exclude(pk=self.instance.pk)` користувач не зможе зберегти профіль, навіть не змінюючи пошту: форма знайде «дубль» — і це буде він сам.

### Форма рецензії на фільм

Рецензії й коментарі — теж введення, і теж вимагають захисту (спам, порожні повідомлення, накрутка оцінок):

```python
class ReviewForm(forms.ModelForm):
    class Meta:
        model = Review
        fields = ["rating", "text"]

    def clean_rating(self):
        rating = self.cleaned_data["rating"]
        if not 1 <= rating <= 10:
            raise forms.ValidationError("Оцінка має бути від 1 до 10.")
        return rating

    def clean_text(self):
        text = self.cleaned_data["text"].strip()
        if len(text) < 3:
            raise forms.ValidationError("Рецензія замала — напиши хоча б кілька слів.")
        return text
```

## Реєстраційна форма: усі рівні разом

Складімо все у форму реєстрації — тут працюють одразу всі механізми:

- **HTML5** — на полях стоять `required` та `type="email"` (браузер відсіє порожнє й криве).
- **`clean_email`** — перевіряє **унікальність** пошти в базі (валідація одного поля).
- **`clean`** — перевіряє, що пароль і підтвердження **збігаються** (міжпольова валідація).
- **Пароль** — валідується через `AUTH_PASSWORD_VALIDATORS` (довжина, не надто поширений тощо).

```python
from django import forms
from django.contrib.auth import get_user_model

User = get_user_model()

class RegisterForm(forms.ModelForm):
    password_confirm = forms.CharField(widget=forms.PasswordInput)

    class Meta:
        model = User
        fields = ["username", "email", "password"]
        widgets = {"password": forms.PasswordInput}

    def clean_email(self):
        email = self.cleaned_data["email"]
        if User.objects.filter(email=email).exists():
            raise forms.ValidationError("Ця пошта вже зайнята.")
        return email

    def clean(self):
        cleaned = super().clean()
        if cleaned.get("password") != cleaned.get("password_confirm"):
            self.add_error("password_confirm", "Паролі не збігаються.")
        return cleaned
```

Правило унікальності описане **один раз** — у формі. JS може продублювати його для UX, але істину зберігає сервер.

## Як помилки повертаються користувачу

**Визначення.** Коли `is_valid()` повертає `False`, усі помилки складаються у `form.errors`, і Django сам домальовує їх у шаблоні поряд із відповідними полями.

**Як це працює.** У view нічого спеціального робити не треба — просто повертаєш ту саму форму назад у шаблон, і вона «памʼятає» свої помилки та введені значення:

```python
def create_post(request):
    if request.method == "POST":
        form = PostForm(request.POST)
        if form.is_valid():
            form.save()
            return redirect("post_list")
        # якщо НЕ valid — form уже містить .errors, просто рендеримо її знову
    else:
        form = PostForm()
    return render(request, "blog/create.html", {"form": form})
```

У шаблоні виводиш помилки поля поряд із самим полем. `{{ form.non_field_errors }}` показує помилки з `clean()`, що не привʼязані до конкретного поля:

```html
<form method="post">
  {% csrf_token %}

  {{ form.non_field_errors }}

  {% for field in form %}
    <p>
      {{ field.label_tag }} {{ field }}
      {% for error in field.errors %}
        <span class="error">{{ error }}</span>
      {% endfor %}
    </p>
  {% endfor %}

  <button type="submit">Зберегти</button>
</form>
```

**Навіщо.** Користувач одразу бачить, *що саме* не так і *біля якого поля*, — а не порожню сторінку. Помилки з `clean_<field>()` лягають до поля, помилки з `clean()` — у `non_field_errors`.

> 💡 Не хочеш вручну перебирати поля — Django вміє відрендерити всю форму одним викликом: `{{ form.as_p }}`, `{{ form.as_div }}` (рекомендований у Django 6.0) або `{{ form.as_table }}`. Помилки він додасть автоматично.

## Типові помилки / Нюанси

- **Покладатися лише на JS.** Найнебезпечніша помилка: «я ж перевірив у скрипті» — а запит летить повз скрипт через `curl`. Це пряма діра безпеки.
- **Валідувати лише авторизацію.** Форма статті, замовлення, рецензія, обкладинка — усе це введення користувача. Незахищена «другорядна» форма — така сама діра, як незахищений логін.
- **Писати правила окремо в JS і в Python руками.** Дублюєш логіку у двох місцях — і рано чи пізно забуваєш синхронізувати: JS каже «ок», сервер каже «помилка», або навпаки. Нехай форма буде єдиним джерелом.
- **Забути `return` у `clean_<field>()`.** Якщо метод нічого не повертає, поле стає `None` — і дані «зникають» без явної помилки.
- **Перевірку унікальності при редагуванні без `.exclude(pk=...)`.** Користувач не зможе зберегти власний профіль, бо форма прийме його ж за «дубль».
- **Плутати `clean_<field>()` і `clean()`.** Одне поле — `clean_<field>()`; звʼязок між полями (пароль=підтвердження, дати сеансу) — тільки `clean()`.
- **Не показувати серверні помилки.** Форма відхилила дані, а користувач бачить порожню сторінку. Завжди виводь `form.errors` / `non_field_errors` / `messages`.
- **Ручні `if` замість форми.** Перевіряти дані вручну у view, коли для цього є `Form` — це зайвий код і порушення DRY. Спершу форма, потім `is_valid()`.

## Підсумок

- Є **дві лінії** валідації: клієнтська (браузер: HTML5 + JS) і серверна (Django: `Form` / `ModelForm` / view).
- **Золоте правило:** серверна — обов'язкова завжди; клієнтська — лише UX і не замінює її. Ніколи не довіряй даним із браузера — JS вимикається, а запит підробляється (`curl`, Postman, DevTools).
- **Валідуй будь-яке введення**, а не лише логін: стаття, замовлення, профіль, рецензія, файл — усе це форми, усе через сервер.
- **JS для логіна не обов'язковий:** достатньо Django-форми + `authenticate`. **Перевіряти у view — треба завжди**, але «перевірити» = використати форму й `is_valid()`, а не писати ручні `if`.
- **Три рівні:** HTML5 (дешеве) → JS (опційно, UX) → Django-форма — головний.
- **Чотири механізми форми:** валідатори полів (`validators=[...]`) → `clean_<field>()` (одне поле) → `clean()` (звʼязок між полями) → `AUTH_PASSWORD_VALIDATORS` (паролі); файли перевіряй за розміром і розширенням.
- **Best practice:** форма — єдине джерело правил; серверні помилки показуй через `field.errors` / `non_field_errors` / `messages`.
